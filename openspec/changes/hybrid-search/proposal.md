# 提案：日记的混合搜索（ILIKE + 向量检索）

## 背景
目前，首页（Dashboard）和日记本内页（DiaryView）中的日记搜索功能仅依赖于 Prisma 的 `contains` 过滤器，这在底层会被转换为基础的 `ILIKE` SQL 查询。虽然这种方式能够提供精确的子串匹配，但它无法实现真正的“模糊”或语义搜索体验。例如，搜索“伤心”时，无法返回包含“大哭”或“抑郁”的日记，轻微的拼写错误也会导致搜不到任何结果。

由于本项目已经在使用启用了 `pgvector` 扩展的 PostgreSQL，并且为了 AI 聊天功能已经存储了日记的向量（embeddings），因此我们在底层基础设施上已经具备了支持语义搜索的条件。

## 目标
通过结合精确关键词匹配（`ILIKE`）和由 AI 驱动的语义匹配（向量搜索），大幅提升日记的搜索体验。这种混合搜索方案既能确保用户可靠地找到包含精确词汇的日记，又能挖掘出在概念上相关的日记，从而营造出符合“魔法日记本”主题的“联想记忆”的魔法感。

## 提议方案：混合搜索策略
我们将采用“分区分层展示”的混合搜索策略，而不是复杂的算法分数融合（如 RRF）。这种方式更实用、更易于实现，且通常能提供更清晰的用户体验。

1.  **首页（Dashboard）搜索：**
    *   执行 `ILIKE` 精确匹配查询。
    *   并发执行使用 `pgvector` 的向量相似度查询。
    *   在列表顶部优先展示精确匹配的结果。
    *   插入一个视觉分割线（例如："✨ 魔杖感应到了以下相关的记忆..."）。
    *   在分割线下方展示最多 10 条语义匹配的结果（需要排除已经在精确匹配中出现过的日记）。

2.  **日记本内页（DiaryView）搜索：**
    *   保持现有的弹窗交互方式。
    *   在当前日记本的范围内，执行精确匹配和向量搜索。
    *   在弹窗的结果列表中，优先将精确匹配的结果展示在顶部。
    *   在底部追加最多 5 条语义匹配的结果。
    *   （可选）添加微妙的徽章或分割提示，以区分语义匹配项。

## 非目标（不做的事情）
*   实现带有中文分词器（如 `pg_jieba`）的 PostgreSQL 原生全文搜索（FTS）。考虑到 Serverless 数据库的限制以及提议的混合方案已经足够有效，这种做法的投入产出比（ROI）过低。
*   在 `ILIKE` 的布尔值和向量的余弦距离之间进行复杂的数学分数融合（例如倒数秩融合 RRF）。

## 技术可行性与影响
*   **可行性：** 极高。数据库 Schema 已经支持 `pgvector`，并且在 `lib/embedding/search.ts` 中已经存在 `searchRelatedDiaries` 函数。我们只需要将其接入主要的搜索 API 并更新 UI 即可。
*   **性能影响：** 中等。向量搜索需要调用大模型（LLM）的 API 将搜索词转换为向量，这会增加轻微的延迟（几百毫秒）。我们需要确保 UI 有优雅的加载状态处理。
*   **成本：** 用于搜索词向量化的 LLM API 调用会带来极少量的成本增加。
