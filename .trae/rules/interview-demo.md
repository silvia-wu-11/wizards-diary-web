# 面试演示模式（魔法日记本）

本 rule 在**面试演示、排练或优化讲法**时生效；技术栈与日常代码约定见 **project-context**、**fullstack-conventions**，此处只约定「怎么讲、代码怎么更好讲」。

## 想表达什么
- 用这个项目讲清：产品定位、AI 能力、全栈架构、工程实践（TDD、结构、开发体验）。**可讲解性 > 纯性能**；冲突时优先故事好讲、结构清楚。

## 方便讲的代码形态
- **分层清晰**：UI / 业务层（hooks 或 service）/ 数据层（Prisma、Server Actions），便于顺着一条链路讲。
- **命名自解释**：如 `createMagicEntry`、`summarizeDiaryWithAI`、`getUserDiaryTimeline`，面试官看名字能跟上。
- **意图性注释**：设计取舍处用少量注释说明「为什么这样设计」，不写语法细节。
- 给方案时优先「能否顺畅讲出去」：小函数、设计处附 1～2 句可对面试官说的解释；多方案时标出更适合面试展示的。

## 建议的讲解顺序
- 先有一条完整「幸福路径」：登录 → 写日记 → 触发 AI 能力（总结 / 打标签 / 魔法图等）。
- 排练时按此组织：① 产品故事与目标 ② 整体架构（技术栈见 project-context）及各层职责 ③ 一条用户路径的代码链路（页面 → API/Server Action → 数据库/AI）④ 工程实践（TDD、错误处理等，具体见 fullstack-conventions）。

## 排练面试时
- 当我说「排练面试」或「帮我优化讲法」时：协助调整命名更口语化；给出简短的「一句话解释」便于记忆（例：「这段把原始日记转成带魔法标签的结构化数据」）。
