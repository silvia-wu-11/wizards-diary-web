## Context

当前日记条目已支持文本、标签、图片三类内容，并已建立从 `Dashboard`、`DiaryView` 到 Zustand、Server Action、Prisma、Supabase Storage 的完整保存链路。此次变更需要在不引入语音转写的前提下，为日记新增“单段录音附件”能力，并同时覆盖首页创建与详情页新建/编辑两个入口。

该变更具有以下约束：
- 一篇日记仅允许一段录音，最长 120 秒。
- 首页录音区位于正文与图片区之后；按钮位于“添加图片”按钮左侧。
- 详情页仅在新建态与编辑态展示录音区，且放置在左页图片区下方。
- 删除录音必须经过与删除日记本一致风格的确认弹窗。
- 录音名称只作为展示元数据保存，不参与真实存储路径命名。

## Goals / Non-Goals

**Goals:**
- 在首页与详情页提供一致的单段录音录制、预览、删除与展示名称编辑能力；详情页已录音态仅保留“删除录音”按钮（需删除后才能重新录音）。
- 扩展日记条目的持久化模型，使录音文件与展示名称、时长、MIME 类型可随日记一并保存。
- 抽取通用确认弹窗基座，复用到删除日记本与删除录音场景，减少重复 UI 结构。
- 将录音交互封装为独立前端组件，降低 `Dashboard` 与 `DiaryView` 的页面复杂度。

**Non-Goals:**
- 不实现录音转文字、关键词检索或向量化接入。
- 不支持一篇日记多段录音、录音分段拼接或波形回放。
- 不修改录音在对象存储中的物理文件名。
- 不引入新第三方录音库，优先使用浏览器原生 `MediaRecorder` 与 `Web Audio API`。

## Decisions

### 1. 使用单段音频字段挂载在 `DiaryEntry` 上，而不是新建子表

选择在 `DiaryEntry` 上直接增加可选字段：
- `audioUrl`
- `audioName`
- `audioDurationSec`
- `audioMimeType`

原因：
- 产品规则已经明确为“一篇日记只需要 1 段录音”，单字段模型足够表达当前需求。
- 现有图片也是以内联字段数组形式挂在日记实体上，前后端改动路径清晰。
- 避免引入新的 Prisma 关联表、额外查询和迁移复杂度。

备选方案：
- 新建 `DiaryAudio` 子表可支持未来多段录音，但会显著增加当前实现与文档复杂度，不适合本次 MVP。

### 2. 录音能力封装为独立客户端组件 `AudioRecorderField`

录音流程需要访问麦克风权限、`MediaRecorder`、`AudioContext`、计时器和本地预览 URL，不适合直接堆叠在 `Dashboard` 与 `DiaryView` 页面内。为此抽取独立组件，内部负责：
- 开始录音、停止录音、超时自动停止
- 录音中的实时声波绘制数据
- 基于 `Blob` 与 `URL.createObjectURL()` 的本地音频预览 URL 与展示名称状态
- 删除录音前打开确认弹窗

页面仅负责接收和提交标准化结果：
- `audioBlob`
- `audioPreviewUrl`
- `audioName`
- `audioDurationSec`
- `audioMimeType`

备选方案：
- 页面内各自实现录音逻辑会导致首页和详情页状态分叉，后续维护成本更高。

### 3. 使用 `Blob` 预览与 `FormData` 直传完成音频保存

录音完成后，前端保留 `Blob` 作为单段录音的唯一源数据，并通过 `URL.createObjectURL(blob)` 生成本地预览地址供试听。用户点击保存日记时，前端将录音包装为 `File` 并放入 `FormData`，通过专用上传入口直传服务端；服务端新增 `uploadAudio()`，将二进制音频上传到独立的音频 bucket，再把返回的公开 URL 写入数据库。

原因：
- 避免把音频转成 `data URL` 带来的体积膨胀与额外编码/解码开销。
- `Blob/File` 更贴近 `MediaRecorder` 原始产物，预览与上传链路都更自然。
- `FormData` 直传更适合文件场景，便于后续补充 MIME、大小和时长校验。
- 未保存的草稿不会产生孤儿音频文件，减少清理成本。
- 首页与详情页可共用同一套提交逻辑。

备选方案：
- 继续复用图片的 `data URL -> Server Action` 方式实现更快，但对音频文件不够高效。
- 停止录音后立即上传可降低保存时延，但会带来草稿回收、失败补偿和重复上传问题。

### 4. 录音名称仅作为展示元数据保存，不参与实际存储路径

用户可编辑录音名称，但真实对象路径仍由服务端统一生成。存储路径继续按用户 ID / 条目 ID / 时间戳生成安全名称。

原因：
- 避免录音重命名触发对象迁移或多次上传。
- 减少服务端路径管理复杂度。
- 满足“可展示和修改名称”的产品诉求即可。

### 5. 通用确认弹窗基座支持按场景注入文案和危险动作

抽取通用 `ConfirmActionModal` 基座，保留当前删除日记本的视觉骨架、遮罩、动画和按钮布局，但将标题、描述、确认按钮文本、图标与危险色作为可配置参数。删除日记本与删除录音分别传入不同文案。

原因：
- 当前删除日记本弹窗已经形成稳定视觉风格，复用它能保持体验一致。
- 录音删除与日记本删除都属于“危险确认”交互，基座化比复制两个近似组件更易维护。

### 6. 详情页录音区只出现在新建态与编辑态的左页图片区下方

详情页延续现有书本双页布局，不强行重排为首页的垂直结构。录音区放在左页图片区与标签区之间，以减少版式破坏；只读态则在详情弹窗或阅读态中展示音频播放器和名称。

原因：
- 满足产品要求，同时保留现有书本式阅读布局。
- 避免大幅调整左右页职责，降低 CSS 与交互回归风险。

## Risks / Trade-offs

- [浏览器兼容性差异] `MediaRecorder` 在不同浏览器上输出的 MIME 类型可能不同 → 通过 `MediaRecorder.isTypeSupported()` 选择可用格式，并在服务端按实际 MIME 处理上传。
- [大文件风险] 用户录满 120 秒时二进制体积仍可能偏大 → 前端限制最长 120 秒，必要时补充大小提示，并在服务端校验 `FormData` 文件大小。
- [未保存丢失] 用户录音后刷新或离开页面会丢失本地录音 → 维持与图片草稿一致的行为，在交互文案中默认接受这一限制。
- [页面复杂度上升] 首页与详情页都增加音频状态 → 通过复用 `AudioRecorderField` 与标准化返回值控制复杂度。
- [确认弹窗重构风险] 删除日记本弹窗改为复用基座可能引发样式回归 → 以现有组件视觉为基准抽象，仅暴露必要配置项，并保留现有默认样式。
- [对象 URL 生命周期] 预览 URL 若不释放会造成内存泄漏 → 在重录、删除录音和组件卸载时主动调用 `URL.revokeObjectURL()`。

## Migration Plan

1. 更新 Prisma schema，为 `DiaryEntry` 增加音频字段并执行数据库迁移。
2. 扩展 DTO、Store 和读取逻辑，保证旧数据在字段缺省时仍可正常读取。
3. 新增基于 `FormData` 的音频上传入口、音频上传封装与存储 bucket 配置说明。
4. 接入首页与详情页 UI，并将删除日记本弹窗切换到通用确认弹窗基座。
5. 通过回归验证已有文本/图片保存、日记本删除与日记详情展示能力未受影响。

回滚策略：
- 若发布后出现严重录音兼容性问题，可仅隐藏前端录音入口；数据库新增字段为可选字段，不影响旧日记读取。

## Open Questions

- 当前没有阻塞实现的未决问题；录音 bucket 名称与大小上限可在实现阶段结合现有 Supabase 约束确定。
