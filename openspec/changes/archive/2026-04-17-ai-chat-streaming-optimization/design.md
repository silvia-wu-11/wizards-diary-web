# AI 对话流式展示优化 · 技术设计

## 设计目标
提升流式展示的及时性与连续性，降低首字延迟与批量渲染带来的卡顿。

## 现状问题
1. **换行依赖**：缓冲区按 `\n` 切分，若 SSE 行被拆分，内容会滞留缓冲区。
2. **自动批处理**：React 18 会合并异步循环中的多次 `setState`，导致更新不够及时。
3. **JSON 解析失败**：流式 JSON 被拆分时会解析失败，已返回内容无法展示。
4. **思考阶段空白**：SSE 仅有 reasoning_content 时，前端无可见反馈。
5. **高频分片卡顿**：单字级分片导致高频渲染，影响流畅度。
6. **缺少指标**：无法量化首字与完成时延来源。

## 优化方案

### 1. 缓冲区增量解析
- 每次收到 chunk 后将其追加到 `buffer`
- 先按 `\n` 解析完整行
- 对剩余 `buffer` 尝试在 `data:` 前缀后做增量解析
- 若仍无法形成完整 JSON，保留在缓冲区等待下一次数据

### 2. 禁用自动批处理
- 使用 `react-dom` 的 `flushSync` 包裹 `setStreamingContent`
- 保证每次可解析内容到达后立即渲染

### 3. 容错解析
- 当 `JSON.parse` 失败时，尝试从字符串中提取 `"content"` 字段的值
- 使用 ES5 正则 `/"content"\s*:\s*"([^"]*)/` 提取可用文本
- 解析到的部分内容直接追加到 `fullContent` 并渲染

### 4. 思考阶段可见反馈
- 当解析到 reasoning_content 且尚无 content 时，展示“思考中”
- 首个 content 到达后切换为实际回复内容

### 5. 高频分片节流合并
- 对流式更新做 30–50ms 的节流合并
- 保持 UI 连续感，同时降低渲染频率

### 6. 流式指标记录
- 记录首分片到首 content 的耗时
- 记录首 content 到完成的耗时

## 影响范围
- 仅影响 `OldFriendChatDrawer` 的流式解析逻辑
- 不涉及 API 路由与后端实现

## 回归点
- SSE 逐字输出是否及时出现
- 断网或接口失败时的错误展示是否正常
- reasoning_content 阶段是否展示“思考中”
- 高频分片是否仍保持流畅
- 指标数据是否能记录完整
